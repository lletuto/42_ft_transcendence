# Technologies Used and Why

## Frontend — Next.js (React, JavaScript)

The frontend is a Next.js application. Next.js was chosen because it provides
routing, server-side rendering and a modern React developer experience out of
the box, which let the team build pages quickly while keeping the UI
component-based and maintainable. Styling is done with a modern CSS solution
(utility classes and component styles) rather than plain global CSS.

## 3D rendering — Three.js and Blender

The game characters and animations were modelled in Blender (modelling, rigging,
weight painting, animation, lighting and camera setup) and exported for the web.
They are rendered in the browser with Three.js, where the lighting and camera
were recreated to match the original scene. Three.js was chosen because it is
the standard WebGL library for real-time 3D in the browser.

## Backend — NestJS (TypeScript)

The backend uses NestJS. NestJS was chosen for its structured, modular
architecture: dependency injection, modules, controllers, guards and a built-in
WebSocket gateway. This made it easy to keep authentication, the game logic and
the chatbot in separate, testable modules. Input is validated server-side with
class-validator DTOs.

## Real-time — WebSockets (Socket.IO)

Live multiplayer matches and the lobby use WebSockets through Socket.IO, which
handles rooms, reconnection and event messaging. HTTP request/response is not
suited to a fast, two-way game, so a persistent socket connection is used for
moves, room creation, rematches and leaving a game.

## Database — PostgreSQL with Prisma

Data is stored in PostgreSQL and accessed through the Prisma ORM. Prisma gives a
typed schema, automatic migrations and a clean query API, which reduced
boilerplate and prevented many SQL mistakes. PostgreSQL was chosen for its
reliability and relational features (users, friendships, messages, games).

## Authentication and security

Passwords are hashed and salted with bcrypt and never stored in plain text.
Sessions use a JWT stored in an httpOnly cookie with refresh-token rotation, and
users can enable two-factor authentication (TOTP) using otplib and a QR code.
All traffic goes through HTTPS, terminated by nginx.

## AI — LangChain, Ollama and Groq

The chatbot uses LangChain to orchestrate a RAG pipeline. Embeddings are
computed locally with Ollama (`nomic-embed-text`) so there is no embedding API
cost, and responses are generated by Groq (`llama-3.3-70b-versatile`) on its
free tier. The vector store is kept in memory and rebuilt at startup.

## Infrastructure — Docker Compose and nginx

The whole stack (database, Ollama, backend, frontend, nginx) is defined in a
single Docker Compose file and started with one command. nginx acts as a reverse
proxy, terminates TLS and routes `/api/` calls to the backend.
