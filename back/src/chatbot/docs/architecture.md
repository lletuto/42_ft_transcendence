# Technical Architecture — ft_transcendence

This document describes how the system is built end to end: the containers, the
request flow, the backend modules, the database, the real-time layer and the AI
pipeline.

## Containers and deployment

The whole project runs in containers orchestrated by Docker Compose and is
started with a single command (`make up`). The services are:

- **db** — PostgreSQL (using the pgvector image), the relational database.
- **ollama** — runs the local embedding model and pulls `nomic-embed-text` on
  startup.
- **backend** — the NestJS API server.
- **front** — the Next.js frontend.
- **nginx** — the reverse proxy and TLS terminator.

nginx is the single public entry point. It serves HTTPS with a self-signed
certificate generated by `make`, routes browser traffic to the frontend, and
forwards API calls to the backend after stripping the `/api/` prefix. This is
why normal usage goes through nginx rather than hitting the backend port
directly.

## Request flow

A typical authenticated request looks like this:

1. The browser sends a request to nginx over HTTPS.
2. nginx routes it: page requests go to the Next.js frontend; `/api/...` calls
   are forwarded to the NestJS backend (prefix stripped).
3. The backend's JWT guard reads the httpOnly cookie, verifies the signature
   and either allows the request or rejects it with 401.
4. A controller receives the request and delegates to a service that holds the
   business logic.
5. The service may read or write the database through Prisma, then the response
   travels back through nginx to the browser.

## Backend modules (NestJS)

The backend is organized into independent modules, each with its controllers
(HTTP entry points) and services (logic):

- **AuthModule** — signup, login, logout, JWT issuing and refresh, 2FA setup
  and verification.
- **UsersModule** — profile, avatar upload, user search, friends.
- **GameModule** — the game gateway (WebSocket) and game endpoints.
- **ChatModule** — private chat history endpoint and chat gateway.
- **ChatbotModule** — the RAG assistant (controller plus the RAG service).

Dependency injection wires services into controllers automatically. Guards
protect routes declaratively, and DTOs with class-validator validate incoming
payloads (for example the chatbot rejects an empty or oversized message before
any processing happens).

## Database (PostgreSQL + Prisma)

Data is relational and accessed through the Prisma ORM. The main models are
User, Friendship (a many-to-many self-relation), Message, Game, GameResult and
WinningMove. Prisma gives type-safe queries, a single schema file and versioned
migrations applied at startup with `prisma migrate deploy`. Because queries are
parameterized through Prisma rather than concatenated SQL, the app is protected
against SQL injection by default.

Importantly, the RAG chatbot does not use the database. Its document chunks and
embeddings live in memory and are rebuilt each time the backend starts.

## Real-time layer (Socket.IO)

Live gameplay and chat use WebSockets through Socket.IO, integrated into NestJS
as gateways. A game is a room with two players. The gateway listens for events
such as creating or joining a room, submitting a move, requesting a rematch and
leaving a game, and it pushes the updated state to both players in the room.
Socket.IO adds automatic reconnection and room management on top of raw
WebSockets, which is essential for a competitive real-time game.

## AI bot (client-side)

The game's AI opponent runs entirely in the browser. It is not a machine
learning model; it is a deterministic predictor with controlled randomness.
Each round it predicts the player's next move by blending frequency analysis,
an order-1 Markov transition model and a win-stay/lose-shift signal, then plays
the counter to the most likely predicted move. It mixes in randomness to stay
human-like, avoids the Well when it predicts Paper from the player (level 2+),
reinforces its own streaks (level 1+) and adapts its aggressiveness to the
score. Because it is client-side it has no server dependency and no latency.

## AI chatbot (RAG pipeline)

The chatbot is a Retrieval-Augmented Generation system implemented in the
backend with LangChain. At startup the RAG service:

1. Reads every `.pdf`, `.txt` and `.md` file in the `docs/` folder.
2. Splits each document into overlapping chunks with a recursive text splitter
   (chunk size 1000, overlap 200).
3. Computes an embedding for each chunk with the local Ollama model
   `nomic-embed-text` and stores the chunk text and its vector in memory.

When a user asks a question:

1. The question is embedded with the same model.
2. Cosine similarity is computed between the question vector and every stored
   chunk vector.
3. The top 4 most similar chunks are concatenated into a context.
4. The context and the question are sent to the Groq-hosted Llama 3.3 70B model
   with an instruction to answer only from the context and always in English.
5. If the answer is not in the context, the model says it cannot find the
   information rather than inventing an answer.

The endpoint is protected by the JWT guard, so only authenticated users can
query the assistant, and the request is validated by a DTO first. Errors from
Ollama or Groq are caught and turned into a friendly message instead of
crashing the request.

## Why these choices

- **In-memory vector store** rather than pgvector: the dataset is curated
  project documentation, and exhaustive cosine search over it stays instant and
  exact. A vector database with an approximate index only becomes worthwhile at
  tens of thousands of vectors.
- **Local embeddings (Ollama)**: free, private and not rate-limited for the
  indexing step.
- **Groq for generation**: a fast, free-tier host for a strong open-weight
  model, giving good answers with low latency.
